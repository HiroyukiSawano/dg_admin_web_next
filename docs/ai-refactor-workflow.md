# dg_admin_web_next AI 重构执行工作流

## 1. 适用场景

用于旧前端迁移、新业务页接入、已有 AI 重构结果二次修正。目标是让 AI 按照当前项目真实规范工作，而不是只按抽象“重构”指令自由发挥。

## 2. AI 输入包

每次让 AI 重构页面时，至少提供以下输入：

- `docs/frontend-framework-spec.md`
- `docs/frontend-visual-baseline.md`
- `docs/refactor-acceptance-checklist.md`
- `src/views/table/tableBasic.vue`
- `src/views/list/listBasic.vue`
- `src/layouts/classicLayout.vue`

如果目标页面是卡片列表或特殊内容页，可再补充最接近的现有参考页，但不替代以上基础输入。

## 3. 两阶段执行

### 第一阶段：结构对齐

目标：

- 目录放置正确
- 路由进入 `src/router/modules`
- 接口进入 `src/services/modules`
- 页面局部状态留在页面内
- 需要的国际化、权限、页面骨架补齐

本阶段要求：

- 先搜索现有实现，再开始改造
- 优先复用现有页面模式和 `Ec*` 组件
- 不处理“美化式大改”，先把结构和约束对齐

阶段输出至少包含：

- 新增或修改了哪些页面骨架
- 复用了哪些现有组件或路由模式
- 是否新增了 service、locale、route 配置

### 第二阶段：视觉对齐

目标：

- 把页面外观调整到当前项目的中后台风格
- 修正间距、按钮区、表格区、分页区、响应式行为
- 移除不符合本项目气质的装饰性写法

本阶段要求：

- 对照 `tableBasic.vue`、`listBasic.vue` 和 `classicLayout.vue`
- 页面样式只能优先写局部 `scoped lang="scss"`
- 优先使用 `var(--el-*)` 和 Element Plus 组件变量

阶段输出至少包含：

- 引用了哪些视觉参考页
- 复用了哪些变量、色彩和布局模式
- 删除或避免了哪些不符合规范的样式写法

## 4. AI 的硬性规则

每次任务都必须包含以下要求：

1. 先搜索现有实现，再写新代码
2. 不新增新的基础 UI 框架、状态框架、请求框架
3. 不新增第二套主题系统
4. 不在页面里直接发裸请求
5. 不为单页新建全局 store
6. 不大量硬编码颜色、圆角、阴影、字号
7. 不绕开现有布局壳、路由 meta、权限模型
8. 输出中必须写明“复用了哪些现有实现”

## 5. 推荐执行顺序

1. 先选一个最差的重构页做试点
2. 先跑第一阶段，评估结构是否合格
3. 再跑第二阶段，评估视觉是否贴近参考页
4. 试点通过后，再批量复制同一套输入包和提示词

## 6. 不建议的做法

- 一句话让 AI “按文档重构整个模块”
- 同时要求 AI 完成结构迁移、视觉重设计、接口协议改造
- 只给抽象文档，不给参考页
- 只看结果页面，不检查 service、route、store、locale 是否跑偏
