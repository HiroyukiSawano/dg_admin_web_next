# dg_admin_web_next 视觉基线与页面模式补充规范

## 1. 目的

本文档补充 `docs/frontend-framework-spec.md` 中未被充分展开的视觉约束，解决“结构符合规范，但页面气质和样式不像当前项目”的问题。

本文件只定义视觉与页面模式，不改变现有框架接口、路由机制、状态管理和服务层约束。

## 2. 视觉事实来源

视觉基线以当前仓库真实实现为准，不以抽象描述为准。后续 AI 或工程师在重构页面时，至少同时参考以下文件：

- `src/views/table/tableBasic.vue`
- `src/views/list/listBasic.vue`
- `src/layouts/classicLayout.vue`

补充参考：

- `src/views/list/listCard.vue`
- `src/styles/app.scss`
- `src/configs/index.js`
- `src/stores/modules/styleStore.js`

## 3. 默认视觉基线

根据当前仓库配置和样式实现，默认视觉基线如下：

- 布局：`classic`
- 主题：`immersive`
- 主色：`#207f4c`
- 视觉效果：`none`
- 圆角：`0`
- 基础字号：`14`
- 字重：`500`
- 字体：优先使用 `GoogleSansFlexVF-Regular`，回退到现有配置中的中文与系统字体

规则：

- 页面应建立在 Element Plus 变量体系上，而不是重造一套颜色、阴影、圆角、字号系统。
- 页面主背景、边框、文字、激活色优先使用 `var(--el-*)`。
- 全局视觉变化应由 `styleStore`、`data-theme`、`data-layout` 和 CSS 变量驱动，不在页面里直接长期修改根节点样式。

## 4. 页面骨架基线

### 4.1 普通中后台页

默认骨架：

1. `el-card` 作为内容容器
2. 查询区放在顶部
3. 工具按钮区紧跟查询区或与查询区同一行
4. 主内容区使用 `el-table`、列表组件或表单内容
5. 分页放在底部

典型参考：

- 表格页：`src/views/table/tableBasic.vue`
- 列表页：`src/views/list/listBasic.vue`

### 4.2 卡片型列表页

卡片型列表可以参考 `src/views/list/listCard.vue`，但仍需保留以下约束：

- 仍属于中后台页面，不转成营销页、落地页或大面积装饰型页面
- 内容优先级清晰，避免复杂叠层和过度装饰
- 卡片 hover 反馈可以存在，但必须克制，不能压过主体信息

### 4.3 页面与布局边界

- 页面的职责是业务内容，不重建全局 header、menu、tabs、breadcrumb
- 布局留给 `src/layouts/*`
- 页面内部只处理局部容器、局部信息层级和局部响应式

## 5. 样式实现规则

### 5.1 应优先采用

- 局部 `style scoped lang="scss"`
- Element Plus 原生组件组合
- `var(--el-color-primary)`、`var(--el-fill-color-blank)`、`var(--el-border-color-*)`、`var(--el-text-color-*)`
- 16px 左右的基础间距节奏，常见值为 `8 / 16 / 24 / 32`
- 清晰的查询区、操作区、内容区分层

### 5.2 尽量避免

- 大量 `:deep`
- 大量内联样式
- 整页手写复杂阴影、滤镜、发光、浮层装饰
- 使用硬编码颜色替代现有主题变量

### 5.3 明确禁止

- 为单个业务页新造一套全局视觉体系
- 自造与当前主题冲突的大面积渐变背景
- 脱离 Element Plus 变量长期硬编码整套配色
- 页面里到处写 `style=""` 取代结构化 SCSS
- 引入新的基础 UI 样式框架
- 为了“看起来高级”而把中后台页改成品牌官网或数据大屏风格

## 6. 常见区域的样式基线

### 6.1 查询区

- 通常位于卡片顶部
- 以 `el-form` + `inline` 为主
- 输入控件宽度保持规律，不追求每个字段都不一样
- 窄屏时允许纵向堆叠，不允许横向溢出

### 6.2 工具按钮区

- 与查询区保持同一套节奏
- 主按钮数量控制在必要范围内
- 批量操作、次级操作通过 dropdown 或次按钮承载

### 6.3 表格区

- 优先用 `el-table`
- 行内操作使用 link button 或 dropdown，避免塞满高饱和按钮
- 状态展示优先沿用现有 `el-tag` 组合方式

### 6.4 分页区

- 位于内容区底部
- 与主内容保持 16px 左右间距
- 桌面端与移动端 layout 可以不同，但视觉密度应一致

## 7. 响应式基线

新增或重构页面至少保证：

- 桌面端信息密度正常，不出现无意义的大留白
- 中小屏下查询区和按钮区可以换行或纵向排列
- 卡片、表格、弹窗不出现横向溢出
- 不依赖固定宽度让页面“刚好能看”

优先延续项目现有做法：

- Element Plus 栅格
- `hidden-md-and-down`、`hidden-md-only` 等类
- 页面内部局部媒体查询

## 8. AI 重构时必须显式声明的复用项

每次重构页面时，AI 输出中必须明确说明：

- 参考了哪个列表页、表格页、布局页
- 复用了哪些 `Ec*` 组件或现有页面骨架
- 使用了哪些现有 CSS 变量或 Element Plus 组件组合
- 哪些样式没有新造，而是沿用当前项目基线

如果 AI 无法列出这些复用项，说明它大概率没有真正贴合本项目风格。
