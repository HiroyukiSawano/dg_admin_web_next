# dg_admin_web_next 前端框架说明与开发规范

## 1. 文档目的

本文档用于约束后续 AI 或工程师在本仓库内进行旧前端重构和新前端开发时的实现方式。目标不是介绍全部 Demo 功能，而是明确这套前端框架的真实组织方式、边界和必须遵守的开发规则。

后续开发默认前提：

- 以当前仓库为唯一事实来源，不自行发明另一套架构。
- 优先复用已有页面模式、布局能力、通用组件和状态模型。
- 新增代码必须融入现有 `Vue 3 + Vite + Element Plus + Pinia + Vue Router + Vue I18n + Sass` 体系。
- 本文档优先服务“可执行开发”，不是宣传型说明。

## 2. 项目定位与总体判断

这是一个中后台前端 Demo / 脚手架项目，不是纯视觉稿，也不是最小化模板。它已经具备以下基础能力：

- 多布局壳：`classic`、`complex`、`dock`、`column`、`mixture`
- 路由模块化组织与动态注入
- 登录、权限、菜单、页签、主题、系统设置等全局状态
- 服务层统一封装
- 国际化框架
- 一批业务型通用组件，命名统一以 `Ec*` 开头
- 列表页、表格页、详情页、弹窗页、结果页、错误页等典型中后台页面模式

因此，后续旧前端重构与新需求开发，应把本项目视为“已有框架 + 业务页面扩展”，而不是从零自由设计。

## 3. 当前技术栈与运行特征

### 3.1 核心栈

- 构建：Vite 6
- 框架：Vue 3，主要使用 `script setup`
- 组件库：Element Plus
- 状态管理：Pinia + `pinia-plugin-persistedstate`
- 路由：Vue Router 4
- 国际化：Vue I18n
- 样式：SCSS
- 图标：Remix Icon、Element Plus Icons
- 图表与复杂能力：ECharts、Vue Flow、TipTap、XGPlayer 等

### 3.2 工程约定

- 使用 `@` 指向 `src`
- 全局样式入口为 `src/styles/index.scss`
- 全局插件入口为 `src/global.js`
- 路由入口为 `src/router/index.js`
- Pinia 入口为 `src/stores/index.js`
- 服务模块入口聚合在 `src/services/index.js`
- 环境变量使用 `VITE_*`

### 3.3 当前仓库现状

- `README.md` 不是项目文档，仍是默认模板，不能作为规范来源。
- 仓库中存在部分中文注释和文案编码异常，阅读时应以代码结构和命名为准，不以乱码文本为准。
- 已尝试执行 `npm run build` 做可构建性验证，但当前环境返回 `esbuild spawn EPERM`，因此不能把“构建通过”视为已验证结论。
- 当前配置 `AUTH_ENABLED = false`，项目默认跳过登录直接进入应用；如后续需要恢复登录，应优先恢复认证开关，而不是新建第二套登录体系。
- 当前工作区已有用户未提交修改：`src/components/EcSituationGuide/index.vue`、`src/components/EcSituationGuide/node.vue`。后续开发不要覆盖这些无关改动。

## 4. 目录职责规范

### 4.1 顶层目录

- `src`：前端主代码
- `mock`：本地/生产 mock 数据入口
- `public`：静态资源和运行时配置

### 4.2 `src` 目录职责

#### `src/views`

放页面级视图。一个业务页面或一个完整的页面模块都应放在这里。

适用内容：

- 列表页
- 详情页
- 表单页
- 仪表盘页
- 页面内专属弹窗组件
- 页面级子路由容器

不应放入：

- 明显跨页面复用的通用组件
- 通用 hooks
- 接口请求函数

#### `src/components`

放可复用组件，当前项目已有大量 `Ec*` 组件。后续开发必须先检索这里是否已存在同类能力，再决定是否新建。

优先复用的典型组件类型：

- 列表展示：`EcListBasic`、`EcListArticle`
- 图表：`EcChart*`
- 业务交互：`EcFilterBar`、`EcTableSelect`、`EcWorkflow`
- 对话类：`EcChat/*`
- 验证/辅助：`BaseVerifySlider`、`BaseError`

#### `src/router/modules`

放业务路由模块。每个模块导出一个路由对象，由 `src/router/index.js` 通过 `import.meta.glob` 自动加载。

新增业务模块时，默认在这里新增对应路由文件，而不是把所有路由堆在路由入口。

#### `src/stores/modules`

放 Pinia 模块。这里只存放跨页面、跨布局、跨会话生命周期有意义的状态。

当前典型模块：

- `authorizeStore`：登录态、用户信息、角色、权限
- `permissionStore`：菜单、动态路由、首页
- `styleStore`：布局、主题、视觉风格、CSS 变量
- `systemStore`：语言、深色模式、水印、自动退出、设备信息
- `tabsStore`：顶部页签

#### `src/services/modules`

放 API 封装函数。页面和 store 只能调用 service，不要在页面中直接写 `axios` 细节。

#### `src/layouts`

放应用壳和布局方案。业务页面不应自己绕过布局系统另起一套页面骨架。

#### `src/styles`

放全局样式、主题样式、媒体适配样式和公共变量。页面私有样式优先写在组件内部 `scoped` 样式中。

#### 其他目录

- `src/hooks`：复用逻辑
- `src/utils`：通用工具
- `src/locales`：国际化文案
- `src/directives`：全局自定义指令
- `src/configs`：系统配置、表格配置等

## 5. 应用启动与全局装配

### 5.1 启动入口

入口文件 `src/main.js` 的职责很清晰：

- 创建 Vue 应用
- 注册 Pinia
- 注册 Element Plus
- 注册 Router
- 注册 I18n
- 注册全局插件 `global`
- 挂载到 `#app`

这意味着后续开发不要在入口层堆业务逻辑。通用能力应放到插件、store、router guard 或布局层。

### 5.2 全局插件 `src/global.js`

全局插件已统一注册：

- 全局配置：`$configs`
- 权限方法：`$auth`、`$role`
- 翻译方法：`$translate`
- 全局指令：`v-auth`、`v-auths`、`v-auths-all`、`v-role`、`v-time`、`v-copy`
- 全量图标组件

规则：

- 若能力需要全局复用且和框架层直接相关，可考虑放入 `global.js`
- 纯业务能力不要塞进 `global.js`
- 页面内能局部引入的，不要强行做全局注册

## 6. 配置体系规范

配置入口为 `src/configs/index.js`，它集中定义：

- 应用名、版本、Logo、版权
- API 地址与应用编码
- token、cookie、storage 相关参数
- 路由白名单、路由来源、角色控制开关
- 系统默认语言、深色模式、水印、自动退出
- 布局、主题、主色、面包屑、tabs 等样式策略

后续开发规则：

- 全局行为开关放配置，不放页面硬编码
- 默认值必须优先通过配置表达
- 不要在多个页面复制同一份业务常量，若属于系统级配置，应进入 `configs`
- 生产环境运行时配置依赖 `window.publicConfig`，因此不要写死只能在编译期替换的值

## 7. 布局体系规范

### 7.1 布局入口

应用主壳在 `src/layouts/index.vue`。其职责：

- 根据 `styleStore.layout` 选择布局组件
- 挂载移动端菜单和系统设置面板
- 管理全局水印
- 响应自动退出
- 初始化布局、主题、视觉效果等样式属性

### 7.2 布局选择规则

后续新增业务页面默认接入当前布局体系，不单独写新的页面壳。只有以下情况才考虑新布局：

- 与现有五种布局无法兼容
- 需求是全局级壳改造，而非单页特例
- 有明确产品设计要求影响整个应用导航结构

### 7.3 页面和布局的边界

页面负责业务内容，布局负责：

- 顶部、侧边、菜单、页签、面包屑
- 移动端适配容器
- 全局视觉效果
- 全局通知、水印、系统级设置面板

不要在单个页面里重造菜单、页签或全局主题切换。

## 8. 路由组织规范

### 8.1 路由层次

本项目实际存在三层路由来源：

1. 常量路由 `constantRoutes`
2. 前端静态业务路由 `frontendRoutes + src/router/modules/*.js`
3. 后端菜单转换路由 `permissionStore -> convertBackendMenu`

### 8.2 常量路由

固定保留：

- 根布局 `/`
- 登录 `/login`
- 错误页 `/403`、`/404`、`/500`
- 重定向页 `/redirect/:path(.*)`
- 兜底 404

这些路由属于框架层，不作为普通业务页面的放置区。

### 8.3 业务路由文件写法

新增模块路由时，遵循现有 `src/router/modules/*.js` 模式：

- 每个文件导出一个默认路由对象
- 顶层一般是目录型路由
- 子路由挂具体页面
- 页面组件使用动态导入

参考模式：`src/router/modules/tableRoutes.js`

### 8.4 路由 `meta` 规范

后续开发必须沿用现有 `meta` 模型，不要另造字段名。常用字段如下：

- `title`：页面标题或 i18n key 入口
- `locale`：国际化菜单 key
- `icon`：菜单图标
- `type`：路由类型，常见为 `menu`、`directory`
- `hidden`：是否隐藏菜单
- `active`：用于隐藏页高亮其父菜单
- `affix`：是否固定到 tabs
- `fullpage`：是否全屏页
- `order`：排序
- `roles`：角色控制
- `hiddenBreadcrumb`：是否隐藏面包屑

规则：

- 菜单页必须有 `meta.type = 'menu'`
- 目录页必须有 `meta.type = 'directory'`
- 弹窗详情、编辑页、隐藏子页要配合 `hidden` 和 `active`
- 需要顶部固定的首页类页面可使用 `affix`
- 不要在页面内部自行控制菜单高亮，应通过路由元信息解决

### 8.5 动态路由机制

`permissionStore.GenerateRoutes()` 的实际流程：

1. 获取菜单来源
2. 与前端静态路由合并
3. 格式化标题，兼容国际化
4. 按角色过滤路由
5. 过滤隐藏菜单
6. 扁平化路由并生成面包屑链
7. 注入到 `Root` 路由下

因此后续如果对接真实后端菜单，应该兼容这里的转换逻辑，而不是绕开 `permissionStore` 另写一套路由注入。

## 9. 菜单、权限与认证规范

### 9.1 认证状态

`authorizeStore` 管理：

- `token`
- `user`
- `roles`
- `permissions`

并负责：

- 登录状态写入
- 登出清理
- token 检查
- 缓存清理

### 9.2 权限模型

项目当前同时支持：

- 角色权限：路由级控制
- 指令/方法权限：按钮、局部操作级控制

权限入口：

- 方法：`permission`、`permissionRole`
- 指令：`v-auth`、`v-auths`、`v-auths-all`、`v-role`

规则：

- 页面内按钮显隐优先使用现有权限方法或指令
- 菜单可见性优先通过路由 `meta.roles`
- 不要把权限判断散落成硬编码 `if (user.xxx)`

### 9.3 登录守卫

路由前置守卫负责：

- 检查 token
- 白名单放行
- 未登录跳转登录
- 首次构建动态路由
- 根路径重定向首页

因此页面层不应重复做登录态重定向。

## 10. 状态管理规范

### 10.1 什么状态进入 Pinia

只有下列状态才应进入 store：

- 跨页面共享
- 跟布局/导航/权限/系统设置直接相关
- 需要持久化
- 多个模块都依赖的全局上下文

### 10.2 什么状态留在页面内部

以下内容默认留在页面组件：

- 查询表单
- 表格当前选中项
- 列表 loading
- 弹窗显隐
- 当前页局部数据
- 页面临时编辑态

不要把每个列表页的查询参数都塞进全局 store。

### 10.3 持久化规则

项目已使用持久化插件。现有持久化存储方案：

- token / user：Cookie
- 角色、权限、样式、系统设置：本地存储

新增全局状态前先判断：

- 是否跨刷新仍需保留
- 是否包含敏感信息
- 是否应该存 Cookie 还是 LocalStorage

没有明确理由，不要默认持久化所有 store。

## 11. 服务层与接口接入规范

### 11.1 接口唯一入口

接口统一走：

- `src/utils/http.js`
- `src/services/modules/*.js`

页面、组件、store 不应直接创建新的 `axios` 实例。

### 11.2 `Http.request` 使用约定

服务函数一般形如：

```js
export const apiSomething = (params) => {
  return Http.request({
    url: '/path',
    method: 'POST',
    data: params,
  })
}
```

统一行为包括：

- 自动挂 token
- GET 默认加时间戳避免缓存
- 统一超时与默认请求头
- 统一处理认证异常

### 11.3 响应契约

现有拦截器逻辑说明：

- 当响应体 `code === 1`，直接返回 `data`
- 当 `status` 为 `401/403/410`，视为认证异常并触发登出
- 其他 `200/300` 会返回 `{ data, msg, status }`

后续开发要求：

- 页面不要各自重复解析同一类响应格式
- 若新后端接口格式不同，应优先在服务层或 `http.js` 中适配
- 不要在页面内部散落大量 `res.code === ...`

### 11.4 服务层边界

服务层负责：

- 发请求
- 输入输出适配
- 与接口协议相关的转换

页面负责：

- 触发时机
- loading 状态
- 表单联动
- 视图展示

## 12. 页面开发模式规范

### 12.1 页面基本写法

默认页面写法：

- 使用 `script setup`
- 使用 `defineOptions({ name: 'PageName' })`
- 以 `ref/reactive/computed` 管理页面局部状态
- 页面容器通常为 `el-card`

### 12.2 中后台常见骨架

本项目最常见页面骨架是：

1. 查询区
2. 工具按钮区
3. 内容区（表格 / 列表 / 表单 / 卡片）
4. 分页区
5. 弹窗或详情子页

参考页面模式：

- 基础表格页：`src/views/table/tableBasic.vue`
- 列表页：`src/views/list/listBasic.vue`
- 表格操作页：`src/views/table/tableOperate/index.vue`

### 12.3 列表页规范

列表页建议职责拆分：

- 页面自身维护查询表单、分页、loading
- 列表项展示可抽成复用组件
- 高级筛选优先复用 `EcFilterBar`

### 12.4 表格页规范

表格页通常使用 Element Plus `el-table`，但必须保持一致性：

- 查询条件在表格上方
- 行内操作按钮与批量操作分开
- 分页位于底部
- 详情/编辑优先复用弹窗或隐藏子路由页面模式

### 12.5 新增页、编辑页、详情页

项目已提供两种模式：

- 弹窗模式
- 页面子路由模式

若是详情/编辑属于同一业务模块，优先采用以下方式之一：

- 简单交互：弹窗
- 表单复杂或需要独立 URL：隐藏子路由页

隐藏子路由页要通过 `meta.hidden + meta.active` 维持菜单高亮。

## 13. 组件复用规范

### 13.1 组件选择优先级

新增能力时按以下顺序决策：

1. 是否已有 Element Plus 原生能力
2. 是否已有 `Ec*` 组件能满足
3. 是否只需要在现有组件上做轻量扩展
4. 最后才创建新组件

### 13.2 新组件放置规则

- 只在单页使用：放当前页面目录内部
- 跨页面复用：放 `src/components`
- 与某个业务组件强耦合的子组件：放该组件内部目录

### 13.3 新组件命名规则

- 页面私有组件：语义化命名，如 `DialogTableOperateEdit.vue`
- 通用组件：延续 `Ec*` / `Base*` 前缀习惯
- 不使用无意义命名，如 `index2.vue`、`test.vue`

## 14. 样式与视觉规范

### 14.1 样式组织

样式分层规则：

- 全局样式：`src/styles/*`
- 组件私有样式：组件内 `style scoped lang="scss"`
- 主题变量：由 `styleStore` 和 CSS 变量控制

### 14.2 样式写法

后续页面优先：

- 使用 Element Plus 变量
- 使用局部 `scoped` 样式
- 仅在必要时使用 `:deep`
- 保持中后台清晰、规整、信息密度适中的视觉风格

避免：

- 在页面里硬编码整套全局颜色
- 大量内联样式替代结构化样式
- 自行引入与现有主题系统冲突的 UI 体系

### 14.3 主题与布局变量

主题、主色、字体、圆角等由 `styleStore` 统一控制。后续如需全局视觉变化，应优先走：

- `styleStore.UpdateStyle`
- CSS 变量
- `data-layout` / `data-theme` / `data-effect`

不要在单页中直接篡改全局根节点样式作为长期方案。

### 14.4 响应式规范

项目已明显兼顾移动端与桌面端：

- 使用 Element Plus 响应式栅格
- 使用 `hidden-md-and-down` 等类名
- `systemStore` 中维护 `device`

新增页面应至少保证：

- 桌面可读
- 中小屏不溢出
- 查询区与按钮区在窄屏下可换行或纵向排列

## 15. 国际化规范

### 15.1 文案来源

国际化入口在 `src/locales`，当前已分：

- `zh-CN`
- `en`

菜单文案、系统文案、设置文案等都已采用 key 管理。

### 15.2 后续开发要求

- 菜单标题优先写 `meta.locale`
- 面向用户的固定文案优先补充到 locale 文件
- 不要在新页面中大量硬编码中文字符串

允许例外：

- 纯内部 Demo 数据
- 临时调试信息
- 后端直出文案

但正式业务页面应尽量走国际化。

## 16. 存储与安全相关规范

项目已经封装：

- `Storage`
- `Cookie`
- `Session`
- `Crypto`

规则：

- 本地持久化优先复用 `src/utils/storage.js`
- 不要直接散用原生 `localStorage` / `sessionStorage` / `document.cookie`
- 涉及登录态、敏感信息时遵循现有加密与前缀规则

## 17. Mock 与环境配置规范

### 17.1 环境变量

当前项目依赖：

- `VITE_APP_TITLE`
- `VITE_APP_API`
- `VITE_APP_API_URL`
- `VITE_APP_CODE`
- `VITE_APP_PORT`
- `VITE_APP_PROXY`
- `VITE_APP_CONTEXT`

后续新增环境相关行为，应继续使用 `VITE_*`。

### 17.2 Mock 机制

`vite.config.js` 已启用 `vite-plugin-mock`。若业务开发需要 mock：

- 优先放在 `mock` 目录
- 保持与 service 层接口语义一致
- 不要把 mock 数据直接长写在正式页面逻辑中作为长期方案

## 18. 新功能接入标准步骤

### 18.1 新增一个普通业务页面

标准步骤：

1. 在 `src/views/<module>` 下新增页面组件
2. 在 `src/router/modules/<module>Routes.js` 中增加路由
3. 补充 `meta.title`、`meta.locale`、`meta.type`
4. 若页面需要对接接口，在 `src/services/modules` 新增或扩展服务函数
5. 若页面存在跨页共享状态，再决定是否新增 store
6. 若存在公用展示块，再决定是否抽到 `src/components`
7. 补充国际化文案
8. 校验菜单显示、页面高亮、移动端布局和权限显隐

### 18.2 新增一个列表 + 详情编辑业务

推荐顺序：

1. 先落列表页骨架
2. 明确查询参数和分页模型
3. 接 service 层
4. 再决定详情使用弹窗还是隐藏子路由
5. 最后补权限按钮、国际化和样式细节

### 18.3 新增全局能力

只有在下面情况才进入框架层：

- 多个业务模块都会复用
- 明确属于系统设置、导航、权限、布局、全局工具
- 有统一维护价值

否则先作为局部能力实现。

## 19. 旧前端重构时的约束

虽然本文件不提供旧系统到新系统的逐页迁移表，但重构时必须遵守以下规则：

- 先把旧页面映射到本框架现有页面类型：列表、表格、表单、详情、结果、弹窗
- 先找现有 Demo 中最相近的页面作为蓝本，再开始改造
- 不把旧项目的目录结构原样搬进来
- 不把旧项目的请求封装、权限逻辑、状态管理原样复制进来
- 不因为赶工跳过路由 meta、国际化、service 层、权限控制

## 20. 给后续 AI 的硬性规则

后续 AI 在本仓库开发时必须遵守：

1. 先搜索现有实现，再写新代码。
2. 新页面优先复用现有页面模式，不重新设计框架层结构。
3. 所有接口都走 `services` + `utils/http.js`，不要在页面里直接发裸请求。
4. 页面临时状态留在页面，跨页共享状态才进入 Pinia。
5. 新增业务路由必须进入 `src/router/modules` 或现有路由模块，不能把路由散落到页面文件。
6. 用户可见文案优先补国际化 key，不长期硬编码。
7. 主题、布局、暗黑、水印等全局视觉能力只通过现有配置和 store 扩展。
8. 能复用 `Ec*` 组件时，不重复造轮子。
9. 不随意引入新的基础 UI 框架、状态框架或请求框架。
10. 不覆盖用户已有未提交改动。

## 21. 明确禁止的做法

- 在页面中直接写新的 `axios.create`
- 为单个页面新建全局 store
- 把业务接口常量和数据转换散落在多个页面里
- 绕开路由 `meta` 手工维护菜单选中状态
- 为了快速上线直接在页面硬编码权限
- 把可复用组件写死在某个页面里
- 大量复制 Demo 页面后不做抽象，形成新的平行规范
- 在全局样式里覆盖大面积 Element Plus 基础规则，导致已有页面回归风险

## 22. AI 开发完成前自检清单

每次新增或重构页面后，至少确认以下问题：

- 页面是否放在正确目录
- 路由是否补全且 `meta` 是否完整
- 接口是否进入 service 层
- 是否误把局部状态塞进全局 store
- 是否优先复用了现有组件
- 是否存在权限按钮漏控
- 是否补充了国际化文案
- 是否兼容桌面和窄屏
- 是否破坏现有布局、菜单高亮、tabs 或面包屑行为

## 23. 结论

这套 Demo 的核心价值不是单个页面，而是“中后台开发约定”本身。后续开发的正确方式是：

- 继承现有分层
- 复用现有模式
- 在路由、权限、状态、服务、样式、国际化这些框架约束内扩展业务

凡是不符合这些约束的实现，即使页面效果看起来正确，也不算符合本项目规范。
